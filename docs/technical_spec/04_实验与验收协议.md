# EvoGrid 智能体自进化平台：实验与验收协议

版本：1.0  
状态：预注册协议基线  
适用范围：地图校准、连续地形验证、Skill 晋升、Prompt/Skill 对照、环境—Agent 耦合实验

## 1. 目的

本协议防止以下常见错误：

- 地图看起来不同，却没有统计证据证明 H 或 p 生效；
- 通过强制开路破坏渗流分布；
- 用训练 reward 代替 Agent 能力；
- 把 reflection 或详细教学 prompt 当成 Skill；
- 用产生 Candidate 的同一批地图验证 Candidate；
- 只展示正例，不统计负场景误触发；
- 在看到结果后修改验证门槛；
- 多次尝试 Candidate 后只报告成功者。

## 2. 结论等级

| 等级 | 最低证据 | 允许表述 |
| --- | --- | --- |
| E0 工程可运行 | smoke test | “接口和流程可运行” |
| E1 地图可控 | 正式地图校准 | “参数改变了目标地图统计量” |
| E2 机制有效 | 配对消融、未见 seed | “该 Skill/机制在指定环境族有效” |
| E3 可迁移 | 跨 H/p/size 未见环境 | “该 Skill 在预注册分布外迁移” |
| E4 Skill Set 演化 | 多阶段、多个 seed、能力基准 | “该课程伴随 Verified Skill Set 和能力增长” |
| E5 环境耦合促进 | 固定/随机/自适应课程对照 | “环境调整促进了持续技能获得或保持” |

不得从低等级结果跳到高等级结论。Mock LLM 结果最高只能支持 E0/E1 的流程和统计接口，不支持真实 LLM 自主学习结论。

## 3. Seed 与数据分区

### 3.1 命名分区

必须区分：

- `train`：Agent 交互和 Candidate 产生；
- `gate`：课程阶段升级；
- `verify`：单个 Candidate 晋升；
- `test`：最终能力与论文结论；
- `bootstrap`：统计重采样，仅作用于已保存结果。

实际 seed 由稳定派生函数生成，不依赖简单连续区间，但 manifest 必须保存展开后的 seed 列表。

### 3.2 隔离规则

1. Candidate 的 `source_episode_ids` 只能来自 train；
2. verify 不得与该 Candidate 的 train/source map seed 重叠；
3. curriculum controller 不得读取 test；
4. 最终 test 结果不得反馈到 Skill revision；
5. 修改 Skill 后形成新版本，必须重新 verify；
6. 修改 generator version 后旧 verify 报告不得自动继承。

任何 seed 泄漏都使相关验证报告无效。

## 4. 协议 A：参数化地图校准

### 4.1 研究问题

1. `p_open` 是否稳定控制实际开放率和贯穿概率？
2. topology H 是否改变开放簇的空间组织？
3. terrain H 是否改变粗糙度斑块尺度，同时保持边际粗糙度分布近似不变？
4. size 是否产生预期有限尺寸效应？
5. generator 是否可复现且无方向性伪影？

### 4.2 Smoke 矩阵

仅用于 E0：

```text
size: [64, 128]
topology_hurst: [0.2, 0.5, 0.8]
p_open: [0.50, 0.65, 0.80]
terrain_hurst: [0.2, 0.5, 0.8]
seeds per cell: 5
```

Smoke 输出不得命名为 formal、verified 或 final。

### 4.3 正式矩阵

建议首个 E1 协议：

```text
size: [64, 128, 256]
topology_hurst: [0.2, 0.5, 0.8]
p_open: [0.40, 0.45, 0.50, 0.55, 0.60, 0.65, 0.70, 0.75, 0.80, 0.85]
seeds per cell: 100
solvability_mode: raw
```

为隔离变量：

- topology 校准时固定 terrain H 和资源参数；
- terrain H 校准使用全开放或固定拓扑，不改变 p；
- resource clustering 单独校准，不与 topology H 同时扫描；
- 若算力不足，可以先对 size=64/128 做全矩阵，再对临界邻域做 size=256；删减必须在运行前写入 resolved config。

### 4.4 地图校准门槛

必须全部满足：

**A0 完整性**

- 每个计划参数单元都有预期 seed 数；
- 无重复 `map_id`（完全相同配置和 seed 除外）；
- 无 NaN/Inf；
- manifest 和 diagnostics 可解析。

**A1 可复现性**

- 随机抽取至少 30 个配置/seed，重建后 map hash 完全一致；
- 改变一个子流 label 不得改变其他图层的数组。

**A2 p 控制**

- correlated 模式的开放格数与目标值误差不超过一个格子；
- 同 H、size 下，贯穿概率随 `p_open` 总体非递减；
- 若出现抽样非单调，必须给出置信区间，不得手工平滑原始数据。

**A3 H 可分性**

- H=0.2、0.5、0.8 的经验 H 中位数顺序一致；
- 至少一个预注册空间指标（半变异函数斜率、邻域相关或斑块尺度）在三个 H 组间呈稳定有序变化；
- 边际 roughness 均值差异不得成为主要解释，rank-normalized 场均值应近似一致。

**A4 无明显轴向伪影**

- 横向和纵向相关统计差异必须报告；
- 若差异超过组内随机波动，generator 不得通过正式门槛。

**A5 临界曲线可估计**

- 对每个 H、size 输出 `P_span(p_open)` 和二项置信区间；
- 用预注册插值方法估计 `P_span=0.5` 对应的有限尺寸 `p50`；
- 不把 `p50` 写成无限系统精确临界点。

### 4.5 必须输出的图

1. 同一边际分布下 H=0.2/0.5/0.8 的代表地图；
2. `P_span` 对 `p_open` 曲线，按 H 和 size 分面；
3. 最大连通分量比例曲线；
4. 输入 H 对经验 H；
5. rough patch size 分布；
6. articulation point/bottleneck 统计；
7. 横纵方向相关差异。

## 5. 协议 B：连续地形与环境改造

### 5.1 数值单元测试

配置固定：

```text
base_move_cost = 0.01
roughness_strength = 0.04
cost_exponent = 1.0
road_move_cost = 0.00
build_cost = 0.10
```

预期：

| r | 未修路单次成本 | 单次道路节省 | 回本所需使用次数 |
| --- | --- | --- | --- |
| 0.0 | 0.01 | 0.01 | 10 |
| 0.5 | 0.03 | 0.03 | 4（向上取整） |
| 1.0 | 0.05 | 0.05 | 2 |

浮点比较使用明确容差，不使用字符串四舍五入后的值做断言。

### 5.2 因果链验收

在相同地图和动作序列上比较：

- 未修路；
- 在目标格修路；
- 在从未复用的格修路。

必须观察到：

- 相同 roughness 对应相同移动成本；
- 道路只改变后续通行成本，不改变 dropoff 奖励；
- 被复用道路按实际次数累积 saved cost；
- 未复用道路净收益为负 build cost；
- DIG 改变可达性，BUILD_ROAD 不改变 blocked/open；
- legacy 模式结果与旧测试一致。

### 5.3 性能验收

在 128x128、partial radius=4、固定 200 步 rollout 中：

- 静态 diagnostics 只计算一次；
- 每步 observation 不复制完整 grid；
- Agent info 不含 evaluator-only 字段；
- episode 结束时 audit snapshot 可生成。

本协议优先检查算法调用次数和数据边界，不预设与硬件绑定的毫秒阈值。

## 6. 协议 C：Skill 候选验证

### 6.1 验证单位

验证单位是精确的 `(skill_id, version, spec_hash)`。自然语言名称相同但 hash 不同的 Skill 必须分别验证。

### 6.2 配对设计

每个 verify seed 运行两次：

```text
A: 相同 Agent + 当前 Verified Skill Set + Candidate disabled
B: 相同 Agent + 当前 Verified Skill Set + Candidate enabled
```

必须固定：

- map seed；
- agent seed；
- primitive action budget；
- LLM action-call budget；
- base prompt hash；
- 已有 Verified Skill Set hash；
- episode max steps。

Candidate proposal/analysis 使用的额外 LLM 调用单独统计，不与执行调用混在一起。

### 6.3 最小样本

- Smoke：每个 stratum 5 个配对 seed，仅验证流程；
- 正式：每个 stratum 至少 30 个配对 seed；
- 至少包含一个目标正场景和一个负场景；
- 声称跨分布迁移时，至少包含 3 个未见环境 strata，总配对样本建议不少于 90。

### 6.4 指标

每个 Skill 在 proposal 时预注册：

- `primary_metric` 和方向；
- `success_condition`；
- `activation_condition`；
- `negative_context_definition`；
- `min_effect`；
- `max_false_trigger_rate`；
- transfer strata。

通用指标：

- paired primary delta；
- success rate；
- activation rate；
- false trigger rate；
- invalid action rate；
- environment action cost；
- LLM call count；
- runtime failure rate；
- transfer delta；
- interaction with existing Skills。

### 6.5 正式晋升 Gate

默认 gate 如下；项目可以在运行前通过版本化协议收紧，但不得看结果后放宽：

**G0 数据完整性**

- 配对样本齐全；
- seed 无泄漏；
- prompt、config、Skill Set hash 一致；
- runtime 无未解释异常；
- 输出 manifest 完整。

**G1 有效性**

- primary paired delta 的双侧 95% bootstrap CI 下界大于 0（maximize 指标）；
- 或 minimize 指标的 CI 上界小于 0；
- 平均效果达到预注册 `min_effect`。

**G2 可靠性**

- 正场景成功率达到预注册门槛，默认不低于 0.60；
- runtime failure rate 不高于 0.02；
- 非法动作率不得显著高于 disabled 条件。

**G3 负场景安全性**

- false trigger rate 不高于预注册门槛，默认 0.10；
- 负场景 primary harm 不得超过预注册容忍值；
- 不得通过“几乎从不触发”同时伪造低误触发和高准确率，必须报告 activation coverage。

**G4 迁移**

- 若 Skill 声称通用，必须在至少两个未见 strata 上方向一致；
- 若只在特定 domain 有效，必须把适用 domain 写入 precondition，不能声称全局迁移。

**G5 非冗余性**

- 在已有 Verified Skill Set 上加入 Candidate 后仍产生独立增益；
- 若效果完全被现有 Skill 覆盖，应进入 merge/revision，不增加 verified skill count。

所有适用 gate 均通过才可晋升。仅 G1 通过不足以晋升。

### 6.6 多 Candidate 控制

同一批轨迹提出多个 Candidate 时：

- 所有 Candidate 都必须记录，包括 schema 失败和效果失败；
- efficacy 检验按 proposal batch 报告多重比较控制，默认使用 Benjamini-Hochberg FDR 0.05；
- 不得只展示通过者而隐藏候选总数；
- revision 后的新版本属于新的检验单位。

### 6.7 验证结局

| 条件 | 状态 |
| --- | --- |
| G0 失败 | verification invalid，修复流程后重跑 |
| G1 失败且方向不稳定 | rejected |
| G1 有趋势但覆盖/安全失败 | revision_required |
| 与旧 Skill 冗余 | revision_required/merge |
| 全部 gate 通过 | verified |

## 7. 协议 D：Prompt 与 Skill 区分实验

### 7.1 实验组

必须至少包含：

| 组别 | 说明 |
| --- | --- |
| No Skill | 固定基础任务 prompt，无 Skill runtime |
| Prompt-only | 详细策略写入 prompt，但不持久化 Skill |
| Handcrafted Skill | 基础 prompt + 人工 RoadLearning Skill |
| Self-proposed Candidate | 基础 prompt + 未验证 Candidate 的受控尝试 |
| Self-proposed + Verified | 基础 prompt + 仅 Verified Skill Set |
| Prompt + Skill | 详细 prompt + Verified Skill，作为参考上限 |

### 7.2 公平性

除了研究因素外，各组必须共享：

- map/agent seed；
- 环境参数；
- episode 和 step budget；
- action-space；
- 基础模型；
- temperature 和采样配置；
- action 阶段 LLM 调用上限；
- fallback policy。

Prompt-only 的详细内容必须在运行前冻结并保存 hash。Self-proposed+Verified 组不得把 Skill procedure 重新全文写入 prompt；否则该组变成 Prompt+Skill。

### 7.3 Skill 与 Prompt 的判据

要声称“形成了 Skill，而不只是 prompt 生效”，至少需要：

1. Skill 以结构化可执行对象跨 episode 保存；
2. 在没有详细教学 prompt 时仍可调用；
3. 在未见 map seed 上优于相同 Agent 的 Skill-disabled 条件；
4. 删除该 Skill 后性能下降，重新加载后恢复；
5. Skill 的触发和执行 trace 可重放；
6. 至少在声明 domain 内迁移；
7. 负场景误触发满足门槛。

如果只有 Prompt-only 提升，结论应写为“详细策略提示改善了表现”。如果 Verified Skill 与 Prompt-only 相当，可以声称 Skill 实现了持久复用，但不能声称它性能更强。

## 8. 协议 E：Capability Benchmark

### 8.1 固定任务族

首个能力基准建议包含：

| 任务族 | 主要能力 | 主要指标 |
| --- | --- | --- |
| T1 局部探索 | frontier exploration、记忆 | 首次发现资源步数、覆盖效率 |
| T2 重复运输 | 复用路线、修路经济性 | 交付量、运输成本、road net payoff |
| T3 负修路场景 | 抑制误触发 | false trigger rate、净收益 |
| T4 临界瓶颈 | bottleneck、DIG | 可达任务成功率、dig cost |
| T5 大地图多矿区 | 区域记忆、层级规划 | 单位步数交付、区域覆盖 |
| T6 分布扰动 | retention、迁移 | shift 前后能力保持率 |

任务族的参数范围、seed 和权重必须在查看最终 test 前冻结。

### 8.2 标准化

每个指标使用预注册 reference/floor 标准化。方向为 maximize 时：

```text
normalized = (value - floor) / (reference - floor)
```

minimize 指标使用相反方向。原始 normalized 值必须保存，可以小于 0 或大于 1；绘图时若裁剪，必须标注且保留原始数据。

### 8.3 Set-level 能力

在每个课程 checkpoint 冻结当前 Verified Skill Set，并在相同 capability test 上运行：

- 全 Skill Set；
- 全部 Skill disabled；
- 逐个 leave-one-skill-out；
- 必要时的关键 Skill 组合消融。

输出：

- `verified_skill_count`；
- `capability_vector`；
- `capability_score`；
- `skill_coverage_matrix`；
- `retention_score`；
- 置信区间。

## 9. 协议 F：环境—Agent 耦合曲线

### 9.1 横轴

横轴使用 `environment_stage_id` 或 `curriculum_checkpoint`。每个点必须能还原：

```text
(size, p_open, topology_H, terrain_H, resource_distribution, disturbance)
```

不得只写一个无法解释的“环境复杂度”。如果需要标量复杂度，必须同时提供原始参数路径。

### 9.2 纵轴

主图至少包含两条量：

- `verified_skill_count`：单 Agent/run 可呈阶梯；
- `capability_score`：固定 benchmark 上的能力。

辅助图包含 Skill coverage、false trigger 和 retention。

### 9.3 阶梯解释

一次上升只能表示一个新版本 Skill 已通过 verify 并进入当前 Skill Set。以下事件不得让阶梯上升：

- 新增 reflection；
- prompt 变长；
- Candidate 被提出但未验证；
- 同一 Skill 重复验证；
- 与旧 Skill 完全冗余的别名；
- 训练 reward 短期提高。

多个 Agent/run 求均值后，离散晋升发生在不同时间，均值曲线可能平滑。必须保留单 run 阶梯图，不能只画平均线。

### 9.4 环境促进自进化的判据

要支持“某环境分布促进持续自进化”，至少比较：

- 静态单环境；
- 预定义阶段课程；
- 参数随机课程；
- 自适应课程（M6）；
- 过快或过难课程。

主要结果：

- 单位训练预算获得的 Verified Skill 数；
- 最终 capability score；
- 新 Skill 的非冗余率；
- retention；
- proposal-to-verification 转化率；
- 计算与 LLM 调用成本。

仅观察到环境和能力同时上升，不足以证明环境变化导致能力上升。

## 10. 统计与报告规范

1. 同时报告样本量、均值/中位数、离散程度和置信区间；
2. 地图 seed 是主要独立单位，不能把同一地图的每一步当独立样本；
3. 配对实验使用 paired delta；
4. bootstrap 默认 10,000 次，并记录 bootstrap seed；
5. 比例使用二项置信区间；
6. 多 Candidate 报告 FDR 控制；
7. 不删除失败 episode，除非预注册为基础设施错误，并报告删除数量和原因；
8. 超时、API 失败和 fallback 必须作为结果字段；
9. 所有图从已保存的原始结果生成；
10. 图表标题或正文必须区分 smoke、pilot、formal。

## 12. R1-R4 验收摘要

### R1 地图统计版

- A0-A5 全部通过；
- p-span、H 校准和代表地图齐全；
- raw 地图没有人工开路；
- 相同 seed 可复现。

### R2 连续地形版

- 数值表与环境结果一致；
- road payoff 可手算复核；
- partial obs 无泄漏；
- 每步不重算全图诊断。

### R3 Skill 基础设施版

- 自然语言不能直接执行；
- Candidate 无法自行晋升；
- Skill 可禁用、重载、重放；
- 正例 fixture 可通过，负例 fixture 被拒绝。

### R4 耦合实验版

- 六组 Prompt/Skill 对照完成；
- 单 run 阶梯和 set-level capability 同时输出；
- 晋升事件可追溯到环境、轨迹和验证报告；
- 结论严格落在 E0-E4 中实际达到的等级。
