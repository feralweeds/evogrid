## 2. 最新路线判断

当前项目路线总体没有走偏，但研究表述必须收敛到“环境-Skill 同步演化”，而不是生物式繁殖或种群进化。

更准确的研究对象是：

```text
环境分布变化
-> 暴露 Agent 当前能力缺口
-> Agent/LLM 提出或修订 Candidate Skill
-> 独立真实环境评价
-> Verified Skill Set 更新
-> Agent 可处理的环境范围扩大
-> 环境课程继续变化
```

当前不能宣称：

- Agent 已经自主形成有效 Skill；
- Verified Skill Set 已经持续增长；
- 环境与 Skill 已经发生同步演化；
- handcrafted positive fixture 等价于真实 EvoGrid rollout 结论。

当前可以宣称：

- 已经建立更可信的 Candidate Skill 提出、执行、拒绝、聚合反馈和修订基础；
- R4 证明了无效 Candidate 可以被拒绝，人工正例可以通过合成晋升通道；
- 下一步应补齐最小通用 DSL 表达能力，并在真实环境中验证人工正例和有限 LLM 回归。

## 3. Prompt 与 Skill 的执行边界

后续实现必须持续区分 Prompt 和 Skill。

| 概念 | 工程含义 | 是否计入能力增长 |
| --- | --- | --- |
| Prompt | 当前回合给 LLM 的外部上下文或策略提示 | 不计入 Verified Skill Set |
| Reflection | 从轨迹得到的文本总结，通常进入后续 Prompt | 不计入 Verified Skill Set |
| Candidate Skill | 结构化、可执行、可哈希、尚未验证的能力假设 | 不计入正式能力，但必须保留审计记录 |
| Verified Skill | 通过独立 verifier 和 registry gate 的 Skill 版本 | 计入 Skill Set 与能力阶梯 |

判断一个能力是否是 Skill，最低要求是：

1. 它以结构化 DSL 存储，而不是只存在于 prompt 文本中；
2. 删除详细教学 prompt 后仍可被 runtime 调用；
3. 在未见 seed 上优于 skill-disabled 条件；
4. 其触发、执行、失败和晋升证据可重放；
5. 负场景误触发满足预注册门槛。

## 4. R4 状态复核

R4 的目标是加固认证基础设施，不是调出一个已经成功自进化的修路 Agent。

R4 已建立或修复：

- DeepSeek candidate-only revision loop，版本从 `1.0.0` 迭代到 `1.0.9`；
- SkillAgent route/memory context bridge；
- continuous terrain route rough diagnostics；
- DSL/schema 对 feature、operator、enum 比较的合法性约束；
- no-op revision guard；
- 聚合失败反馈，不泄漏 per-seed 明细；
- `max_uses_per_episode`、`stop_after_success` 和 episode state；
- handcrafted positive fixture 的 Candidate -> Verifier -> Registry -> Verified 合成晋升通道。

R4 仍未证明：

- DeepSeek 自主提出的 road Skill 有效；
- 真实 EvoGrid rollout 中修路能提升任务表现；
- Skill Set 发生了持续增长；
- 环境课程促进了自进化。

讨论记录中报告 R4 完整测试为 `306 tests OK`。后续执行新工作包前仍必须重新运行相关测试；不得把讨论记录中的测试结果当作当前工作区自动有效。

## 5. 当前优先级变更

`05` 文档曾建议优先恢复 R1 正式地图校准。该建议仍适用于正式论文级地图统计，但 2026-07-22 讨论后，当前工程主线应调整为：

1. 完成 R5 最小通用路线目标选择 DSL；
2. 用真实 EvoGrid rollout 跑通人工 road positive skill；
3. 再进行有限次数 LLM Candidate 回归；
4. 暂停道路专用扩张，尽快接入第二类 Skill，例如 DIG；
5. 在至少两类 Skill 可评价后，再进入环境族矩阵实验。

理由：

- R4 的主要瓶颈已从 verifier 是否能晋升，转为 Candidate 是否能表达“少量、有序、高收益地行动”；
- 继续 v10/v11 的 DeepSeek predicate 微调信息增量低；
- 直接扩展道路 oracle 会把答案写进平台，破坏 Skill 形成问题；
- 只有同一套 DSL 能支持 road 和 DIG，平台才像通用 Skill 系统。

## 6. R5 总目标

R5 的目标是提供最小、通用、可审计的目标选择能力，而不是替 Agent 计算修路最优解。

R5 允许提供：

| 类型 | 允许内容 |
| --- | --- |
| 可观测候选 | 当前 episode 中 Agent 已观察到的 route tiles、局部地形档位、访问次数、是否已有道路 |
| 通用操作 | filter、rank、select first、episode target memory、有限预算 |
| 通用估计 | 基于已观察历史和 route/memory 的可审计估计器 |
| 执行控制 | `max_uses_per_episode`、`stop_after_success`、目标锁定与 trace |

R5 禁止提供：

| 类型 | 禁止内容 |
| --- | --- |
| 隐藏真值 | 全图最短路、隐藏矿点、未见 tile roughness、test split 标签 |
| 未来 oracle | 真实未来经过次数、真实最终净收益、预先算好的最佳修路位置 |
| 道路专用答案 | `find_profitable_road_tile`、`oracle_road_payoff`、只服务 BUILD_ROAD 的特殊函数 |
| final feedback | 使用 final/test 结果指导 Candidate revision |

## 7. R5 工作包

### WP-R5-00：执行基线复核

目标：确认当前 R4 代码和文档状态可作为 R5 起点。

输入：

- `06_R4_Postmortem_and_Positive_Fixture.md`
- `docs/EvoGrid_今日讨论总结_2026-07-22.md`
- 当前 `evogrid/skills/`
- 当前 `tests/`

必须完成：

1. 记录当前 git status，不清理用户无关改动；
2. 运行完整测试；
3. 运行 handcrafted positive fixture；
4. 确认 R4 stop rule 仍成立；
5. 写出本次 R5 baseline run manifest。

退出门槛：

- 完整测试通过，或已明确区分既有失败；
- positive fixture 仍能通过；
- 没有把 fixture 结论写成真实 rollout 结论。

### WP-R5-01：可观测 Route Candidate Source

目标：向 SkillContext 提供 route-local candidate tiles，但只包含 Agent 可观察信息。

修改范围建议：

- `evogrid/skills/context.py`
- `evogrid/agents/skill_agent.py`
- route planner 或 memory summary 相关模块
- `tests/test_skill_context_leakage.py`
- 新增 `tests/test_route_candidate_context.py`

数据契约建议：

```json
{
  "route": {
    "exists": true,
    "is_known_transport_route": true,
    "observed_tiles": [
      {
        "pos": [12, 8],
        "terrain_band": "ROUGH",
        "tile_type": "open",
        "has_road": false,
        "visit_count_bucket": "medium",
        "distance_from_agent": 3,
        "route_order": 0
      }
    ]
  }
}
```

约束：

1. `observed_tiles` 只能来自已观察 route 和 memory；
2. 不得包含 hidden shortest path、future use truth、oracle payoff；
3. 坐标若来自 memory，必须能在 trace 中解释其来源；
4. candidate 数量应有上限，防止 prompt/runtime 膨胀。

退出门槛：

- leakage 测试证明 evaluator-only 字段无法进入 SkillContext；
- route candidate 顺序稳定；
- 无 route 时返回空候选而不是伪造目标。

### WP-R5-02：最小通用 SELECT_TARGET DSL

目标：实现一个可复用于 road 和 DIG 的目标选择操作。

建议 DSL：

```json
{
  "op": "SELECT_TARGET",
  "source": "route.observed_tiles",
  "filters": [
    {"feature": "candidate.has_road", "op": "eq", "value": false},
    {"feature": "candidate.terrain_band", "op": "in", "value": ["ROUGH", "VERY_ROUGH"]}
  ],
  "rank_by": [
    {"feature": "candidate.route_order", "direction": "asc"},
    {"feature": "candidate.visit_count_bucket", "direction": "desc"}
  ],
  "select": "first",
  "store_as": "target"
}
```

必须支持：

- source 白名单；
- candidate feature 白名单；
- filter 与 rank 类型检查；
- 稳定 tie-break；
- 空候选安全返回；
- trace 中记录候选数、过滤数、选中目标 hash。

不得支持：

- 任意表达式；
- 读取全图；
- 根据隐藏收益排序；
- 道路专用 `best_tile` 快捷操作。

退出门槛：

- schema、predicate、runtime 定向测试通过；
- 非法 feature/op/source 被拒绝；
- 同输入可重放。

### WP-R5-03：目标锁定与 Episode State 集成

目标：让 Skill 在一个 episode 内记住已选择目标，并与现有预算控制合并。

必须完成：

1. `SELECT_TARGET` 可以写入 episode-local target；
2. 后续 `ACT` 或 route-following 能引用该 target；
3. `max_uses_per_episode` 对成功环境动作生效；
4. `stop_after_success` 后续 step 不再重复触发；
5. reset 后清空 episode state；
6. trace 能重放 target 选择和使用次数。

退出门槛：

- 同一 episode 不重复建设超过预算；
- 新 episode 可以重新选择；
- 无目标时回退基础策略。

### WP-R5-04：真实环境 Handcrafted Road Skill

目标：用真实 EvoGrid rollout 验证“如果人工写出合理 Skill，当前平台能否在真实环境中测到正效果”。

实验链路：

```text
env.reset
-> Agent 观察并形成 route/memory
-> Skill 选择 route-local target
-> 真实 BUILD_ROAD
-> 后续真实复用道路
-> 记录 road usage 与 road_net_payoff
-> verifier 比较 skill on/off
-> registry 按 gate 决定是否晋升
```

必须比较：

- skill disabled；
- handcrafted candidate enabled；
- verified 后重新加载；
- negative road context。

最低输出：

- `run_manifest.json`
- `episodes/metrics.csv`
- `skills/skill_trace.jsonl`
- `skills/reports/<skill>/<version>/<verification_id>.json`
- road target selection trace
- road usage 与 road net payoff 汇总

退出门槛：

- 指标来自真实 env step，而不是 synthetic fixture；
- positive context 中 road_net_payoff 为正或达到预注册 min_effect；
- negative context false trigger 不超阈值；
- 若失败，保留失败报告并进入 DSL/Skill 修订，不删除 seed。

### WP-R5-05：有限 LLM Candidate 回归

目标：检验新 DSL 是否让 LLM 能提出更有效 Candidate。

约束：

1. 使用基础 prompt，说明 DSL 能力，但不得直接给完整修路答案；
2. 最多允许 2-3 次有实质 executable 变化的 revision；
3. revision feedback 只用聚合指标；
4. 不读取 per-seed 明细和 final/test；
5. 所有失败 Candidate 都进入 Experiment Archive；
6. 继续使用 no-op revision guard。

对照组：

| 组别 | 目的 |
| --- | --- |
| Prompt-only | 测详细提示是否足够 |
| Handcrafted Skill | 测平台上限与正例 |
| Self-proposed Candidate | 测 LLM 是否能形成候选 |
| Self-proposed Verified | 测是否真正进入 Skill Set |

退出门槛：

- 即使 LLM 失败，也要能说明失败在 proposal、schema、runtime、activation、payoff 还是 false trigger；
- 若通过，只能声称在指定环境族形成第一个 Verified Skill，不得扩展到环境-Skill 同步演化。

### WP-R5-06：Road 主线停止审计

目标：防止继续把道路 DSL 做成隐藏答案系统。

必须回答：

1. 新增 DSL 是否可复用于 DIG？
2. 是否加入了道路专用 oracle？
3. Handcrafted 与 LLM Candidate 的差距在哪里？
4. 当前能力提升来自 Prompt、人工规则、DSL 能力，还是 LLM 自主 Candidate？
5. 是否已经满足进入第二类 Skill 的最低条件？

退出门槛：

- 生成 R5 postmortem；
- 明确是否允许进入 `WP-DIG-01`；
- 不再继续无上限 DeepSeek road v10/v11/v12。

## 8. 第二类 Skill：DIG 预备任务

R5 完成后应尽快启动 DIG，而不是继续扩张 road。

### WP-DIG-01：DIG 场景与能力定义

目标：定义“连通修复/瓶颈打通”类 Skill 的正负场景。

建议环境：

| 场景 | 特征 | 期望 Skill |
| --- | --- | --- |
| 低连通瓶颈 | base 与 ore 近但被少量障碍隔开 | 选择性 DIG |
| 高连通开放图 | 已有绕行路径足够便宜 | 避免无谓 DIG |
| 粗糙绕行 vs 挖掘捷径 | DIG 成本和移动成本存在 tradeoff | 成本敏感目标选择 |

输出：

- DIG Candidate Skill objective；
- positive/negative context definition；
- min_effect 与 false trigger threshold；
- 所需 observable candidate source。

### WP-DIG-02：复用 R5 SELECT_TARGET

目标：证明 R5 的目标选择 DSL 不是道路专用。

必须尝试：

- source 使用 local frontier、observed obstacle boundary 或 memory candidate；
- filters 使用 tile_type、adjacent_open_count、distance_from_agent；
- rank_by 使用距离、访问次数、frontier score；
- ACT 使用 `DIG`。

退出门槛：

- 不增加 DIG oracle；
- 与 road Skill 共享 runtime 操作；
- schema 和 trace 能区分不同 Skill 的目标来源。

## 9. 环境-Skill 关系实验的进入条件

只有满足以下条件，才进入环境族矩阵实验：

1. 至少一类 Skill 在真实 rollout 中完成 verified 或明确负结果；
2. 至少第二类 Skill 已定义并跑通 smoke；
3. Prompt-only、Handcrafted、Self-proposed Candidate、Verified Skill 的边界可审计；
4. 环境参数 `p_open`、H、size、资源分布能写入每个 episode manifest；
5. Capability benchmark 至少覆盖 road 正负场景和 DIG 正负场景；
6. mock/smoke 数据不会进入正式图。

首批环境族建议：

| 环境族 | 参数特征 | 观察重点 |
| --- | --- | --- |
| E0 | 高连通、低粗糙度、小地图 | 基础导航与采矿 |
| E1 | 高连通、大块粗糙运输区 | 是否诱导 road Skill |
| E2 | road 正负场景混合 | 是否形成选择性投资 |
| E3 | 低连通、有瓶颈 | 是否诱导 DIG Skill |
| E4 | 大地图、多矿点 | 区域记忆、Skill 组合和迁移 |
