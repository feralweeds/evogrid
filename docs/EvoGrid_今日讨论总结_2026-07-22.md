EvoGrid 今日讨论总结
从“修路技能验证”到“环境—Skill 同步演化”
讨论日期：2026 年 7 月 22 日

| 核心结论<br>当前项目路线总体没有走偏。R4 已经把 Candidate Skill 的提出、执行、独立评价、拒绝、聚合反馈与修订机制搭得更可信；但这仍只是能力评测与认证基础设施，还没有证明 Agent 已经完成自主 Skill 演化。今天最重要的概念修正是：研究重点不是生物式“繁殖”，而是环境分布变化如何诱导 Skill 形成，以及 Skill Set 扩展如何反过来扩大 Agent 可处理的环境范围。 |
| --- |

本文整理今天围绕研究目标、Skill 定义、独立验证、R4/R5 工程进度以及下一步路线的全部关键结论。

## 一、我们一开始真正想研究什么

| 原始研究目标<br>在参数化、可改造、局部可观测的环境中，研究环境分布的变化如何诱导 Agent 自主提出和修订 Skill；经独立评价后，Skill Set 的扩展是否能够扩大 Agent 可处理的环境范围，并支持环境继续变化。 |
| --- |

修路不是最终目标。修路只是第一个受控实验，用来验证“环境改造会产生长期后果，Agent 能否从后果中总结可复用能力”。
- 环境不是固定地图，而是由 p_open、H、地图尺寸、资源分布等参数生成。
- Agent 不只完成一次任务，而要从连续经历中形成可复用的 Skill。
- Prompt 不能冒充 Skill；更长的提示词不等于能力增长。
- Agent 能力不能只用一次 reward 衡量，而要看 Skill Set、泛化、保持和迁移。
- 最终研究对象是“环境变化—能力缺口—Skill 形成—可解环境扩大”的循环。

| 环境分布 Eₜ<br>↓ 暴露当前能力缺口<br>Candidate Skill 产生或修订<br>↓ 真实环境评价<br>Skill Set 更新为 Sₜ₊₁<br>↓ 扩大可解环境范围<br>环境进入 Eₜ₊₁ |
| --- |

## 二、今天澄清的几个核心概念
### 2.1 Prompt 与 Skill 的区别
Prompt 是当前回合给模型的外部说明；Skill 是跨回合、跨任务、跨环境仍能被调用和验证的能力。

| 概念 | 通俗解释 | 判断标准 |
| --- | --- | --- |
| Prompt | 这一次告诉模型应该怎么想、怎么做 | 删除详细提示后，能力通常消失 |
| Skill | Agent 已经形成并能复用的解决办法 | 未见环境、基础 Prompt 下仍然有效 |

### 2.2 Candidate Skill 与 Verified Skill
Candidate Skill 是 Agent/LLM 提出的“能力假设”；Verified Skill 是经过隔离环境评价、达到预先标准后被登记的可靠能力。

| LLM/Agent：提出“我可能学会了什么”<br>↓<br>Candidate Skill：尚未证实的能力假设<br>↓<br>Evaluator / Verifier：在未见环境中测试<br>↓<br>Registry：达到标准后登记为 Verified Skill |
| --- |

### 2.3 “宣布学会”的是不是另一套系统
是，但它不是另一套替 Agent 学习的 AI，而是一套“考试与认证系统”。Agent 负责提出 Skill；Runtime 负责执行；Evaluator/Verifier 负责测量；Registry 负责合法状态更新。

| 关键边界<br>Verifier 可以判断一个 Skill 是否达到可靠能力标准，但不能替 Agent 写答案、不能泄漏单个 seed 的最优动作，也不能使用 final test 指导修订。 |
| --- |

## 三、对“进化”概念的修正
### 3.1 今天提出的疑问
如果只保留通过验证的好 Skill、拒绝所有坏结果，这更像产品验收，而不一定像严谨的进化。进化中间结果可能有好有坏，暂时失败的方案也可能包含有价值的信息。
### 3.2 最终达成的判断
这个疑问有道理，但不必因此引入完整的生物式繁殖、种群、父代选择和交叉变异。你们关注的是环境与 Skill 的同步变化，而不是研究某种 evolutionary algorithm。

| 当前更准确的“自进化”定义<br>Agent 在环境压力下产生或修改 Skill；系统记录这些变化并评价它们；可靠 Skill 进入正式能力库；环境根据当前能力边界继续变化。这里的核心是能力结构随环境发生持续变化，不要求模拟生物繁殖。 |
| --- |

### 3.3 建议采用的五个容器/模块

| 模块 | 保存什么 | 作用 |
| --- | --- | --- |
| Trajectory Store | 环境交互轨迹、成功和失败 | 提供 Skill 形成依据 |
| Candidate Workspace | 当前正在提出和修订的 Skill | 允许好坏结果并存 |
| Experiment Archive | 历史 Candidate、环境、评价和失败原因 | 保留研究证据，避免重复失败 |
| Skill Evaluator / Verifier | 连续评价向量和认证结论 | 测量能力，不负责“繁殖” |
| Verified Skill Set | 达到可靠标准的正式 Skill | 代表 Agent 当前可依赖能力 |

环境—Skill 同步发展的核心控制器应是 Environment Curriculum Controller：它根据当前 Skill Set 的能力边界，选择下一批“既不太简单、也不完全不可解”的环境。
## 四、当前 R4 阶段到底在做什么
R4 的真实目标不是“调出一个会修路的 Skill”，而是加固下面这条闭环：

| LLM Proposal<br>↓<br>Candidate Skill<br>↓ 独立真实评价<br>失败聚合反馈<br>↓<br>Candidate Revision |
| --- |

Coding Agent 实际并行做了两件事：一是检查 DeepSeek Candidate 是否有效；二是修补平台边界，确保坏 Candidate 被正确拒绝，而不是因为上下文、诊断或 DSL 漏洞产生错误结论。
### 4.1 R4 已经完成的工作
- 跑通 DeepSeek revision loop，版本从 1.0.0 迭代到 1.0.9，始终保持 candidate-only。
- 修复 SkillAgent 无法获得 route / memory 摘要的问题，使依赖路线信息的 Skill 可以真实触发。
- 修复连续粗糙度地图的 verifier 诊断，避免把正向修路机会误标成负向场景。
- 加强 DSL/schema：拒绝不存在的 feature、非法操作符和错误的 enum 比较。
- 加入 no-op revision guard：只改版本号、描述或理由而不改执行逻辑的修订会被拒绝。
- 改进聚合反馈：提供激活后的平均建路数、road payoff、road usage 等，但不泄漏 per-seed 明细。
- 新增 R4 复盘文档与 handcrafted positive fixture。
- DSL 新增 max_uses_per_episode 和 stop_after_success。
- Runtime / SkillAgent 增加 episode state，能够跨 step 记住本局已经使用 Skill 的次数。
- Coding Agent 报告完整测试为 306 tests OK，工作区干净。
### 4.2 Handcrafted positive fixture 证明了什么
它是人工写出的已知正例，用来检查“考试系统是否能让一份标准正确答案通过”。结果 decision = verified，说明 Candidate → Verifier → Registry → Verified 的晋升通道可以工作。

| 它没有证明<br>它没有证明 DeepSeek 自主形成了有效 Skill，也没有证明真实 EvoGrid rollout 中修路会提高任务表现。当前 fixture 主要是合成诊断，而不是最终科学结果。 |
| --- |

## 五、目前已经证明与尚未证明的边界

| 已经证明 / 基本建立 | 尚未证明 |
| --- | --- |
| Candidate 不能由 LLM 自行晋升 | DeepSeek 已经自主学会修路 |
| 无效 Candidate 能被拒绝 | 真实环境中存在第一个自主 Verified Skill |
| 失败可以通过聚合反馈继续修订 | Verified Skill Set 已经持续增长 |
| Skill 可以限制每 episode 的使用次数 | 环境与 Skill 已经发生同步演化 |
| SkillAgent 可以跨 step 保存 episode 状态 | 不同 p、H、size 会稳定诱导不同 Skill |
| 人工正例可通过合成晋升通道 | Skill 可以迁移到大地图或新环境 |
| R4 闭环和平台边界更完整 | Agent 自进化已经成功 |

## 六、当前路线是否正确

| 总体判断<br>当前路线仍然正确。R4 是必要的测量与认证基础；R5 也值得做，但只能补“通用 Skill 表达能力”，不能把完整修路答案逐步写进 DSL。 |
| --- |

判断 R5 是否走偏，只需问一个问题：DSL 是在提供通用工具，还是在直接提供修路问题的正确答案？

| 可以提供的通用信息/工具 | 不应提供的隐藏答案 |
| --- | --- |
| 已观察路线、地形档位、历史访问次数、是否已有道路、当前 episode 使用次数 | 未来真实经过次数、全地图最优修路位置、真实最终净收益、隐藏矿点信息 |
| 通用 filter、rank、select、target、budget、episode state | find_profitable_road_tile、oracle payoff、预先算好的最佳目标 |

## 七、接下来最合理的执行顺序
### 7.1 第一优先级：R5-DSL-ROUTE-TARGET-MINIMAL
实现最小、通用的路线目标选择能力：
1. 候选来源只允许使用 Agent 已观察到的当前路线格子。
1. 支持通用 filters，排除不符合条件或已经是道路的格子。
1. 支持基于可观察特征的 rank_by。
1. 支持 select: first 或选择排序后的第一个目标。
1. 目标和使用次数在 episode 内保持。
1. 与 max_uses_per_episode、stop_after_success 集成。
1. 接口必须可以复用于后续 DIG Skill，而非只服务 BUILD_ROAD。
这一阶段不要实现未来使用次数预测、最优道路收益 oracle、完整地图查询或道路专用“最佳格子”函数。
### 7.2 第二优先级：真实环境人工正例
让人工已知合理的 Skill 在真实 EvoGrid 中走完整链路：

| env.reset<br>↓<br>Agent 观察并取得 observed route<br>↓<br>Skill 选择目标并真实 BUILD_ROAD<br>↓<br>后续真实使用道路、计算成本节省<br>↓<br>Verifier 比较有 Skill / 无 Skill<br>↓<br>决定是否晋升 |
| --- |

- 检查是否选中正确路线目标。
- 检查是否只建设规定次数。
- 检查道路是否被后续真实使用。
- 检查 road_net_payoff 是否为正。
- 检查任务总表现是否没有明显下降。
- 检查负场景是否避免错误建设。
### 7.3 第三优先级：有限次数的 LLM 回归实验
人工真实正例通过以后，再让 DeepSeek 使用新 DSL 提出 Candidate。最多允许 2–3 次有实质逻辑变化的修订；继续使用聚合反馈，不读取单 seed 和 final test。
建议比较 Prompt-only、Handcrafted Skill、Self-proposed Candidate 三组，用来判断能力提升来自提示、人工规则，还是 LLM 自主形成的 Skill。
### 7.4 第四优先级：尽快加入第二类 Skill
无论道路 Candidate 是否通过，都不应继续无限扩张道路专用 DSL。完成最小 R5 后，尽快用同一套 Candidate → Runtime → Evaluator → Registry 流程测试 DIG 目标选择。
只有同一套框架能够支持道路和 DIG 等不同能力，才能证明平台是通用 Skill 系统，而不是复杂的修路规则系统。
### 7.5 第五优先级：进入环境—Skill 关系实验
在至少具备两类可评价 Skill 后，开始用环境矩阵回答导师真正关心的问题。

| 环境族 | 参数特征 | 观察重点 |
| --- | --- | --- |
| E0 | 高连通、低粗糙度、小地图 | 基础导航与采矿 |
| E1 | 高连通、大块粗糙运输区 | 是否诱导修路类 Skill |
| E2 | 修路正负场景混合 | 是否形成选择性投资能力 |
| E3 | 低连通、有瓶颈 | 是否诱导 DIG / 连通修复 Skill |
| E4 | 更大地图、多矿点 | 区域记忆、Skill 组合与迁移 |

## 八、当前阶段明确不做什么
- 不引入 Skill 种群、父代选择、繁殖、交叉或 MAP-Elites。
- 不继续无限生成 DeepSeek v10、v11、v12。
- 不继续增加道路专用 oracle、收益预测器和隐藏答案字段。
- 不提前运行 final test partition。
- 不立即推进无限 chunk world、多智能体或大规模结论实验。
- 不因为 fixture verified 就宣布 Agent 自主进化成功。
## 九、建议采用的停止规则

| R5 Stop Rule<br>R5 最多完成：最小通用路线目标选择、一个真实人工正例、一次有限的 LLM Candidate 回归。完成后必须暂停道路主线，检查 DSL 是否能支持 DIG、是否使用了过多人工特征，以及当前进步到底来自 Skill 还是平台替 Agent 计算了答案。 |
| --- |

## 十、当前项目最准确的定位
当前 EvoGrid 不是已经完成的“自进化 Agent”，而是：

| 当前定位<br>一个正在建立严谨 Skill 提出、执行、评价、修订和认证机制的环境塑造型能力演化平台。平台已经具备拒绝伪 Skill 的基础，但尚未得到第一个在真实环境中由 LLM 自主提出并通过验证的 Skill。 |
| --- |

接下来的研究重心应从“继续优化修路规则”转向“接通真实 Skill 闭环，并开始观察不同环境分布如何塑造不同 Skill”。
## 十一、一句话总结

| 今天的最终共识<br>我们不需要把 EvoGrid 做成生物式繁殖算法；我们需要的是一套可审计的环境—Skill 同步发展系统：环境暴露能力缺口，Agent 提出和修订 Skill，评价系统测量其真实效果，可靠 Skill 进入正式能力库，而环境再根据能力边界继续变化。 |
