## 一、导师提出的核心建议

### 1. 把地图扩大，向开放世界发展

导师认为当前 demo 有意思，但地图规模还比较小。可以考虑：

* 使用更大的二维地图；
* 地图不是人工固定设计，而是程序化生成；
* 智能体只能看到局部区域，需要持续探索；
* 环境改造长期保留，例如修过的路、挖开的通道；
* 后续可采用分块生成，而不是一次性加载整张无限地图。

但“大地图”本身不等于开放世界。真正的开放性还应包括：

* 局部观测；
* 持续生成；
* 多区域、多资源；
* 环境变化；
* 技能可以跨区域复用。

---

### 2. 用可调参数控制地图分布

导师建议，不要只生成简单均匀随机地图，而要研究：

> 不同环境分布是否会决定智能体自进化能否持续。

重点考虑：

* 矿石是均匀分布还是聚集分布；
* 地形是小块随机分布还是多尺度聚集；
* 粗糙区域是否形成连续区域；
* 地图连通性和瓶颈结构如何变化；
* 是否可以用少数参数连续控制环境复杂度。

最终环境可以写成参数化生成器：

[
\text{Environment}(\text{size},p,H,\text{resource distribution},\ldots)
]

---

### 3. 引入分形布朗曲面和 Hurst 指数

导师提到可以参考二维分形布朗曲面，用 Hurst 指数 (H) 控制地形结构。

这里的 (H) 不是某一格的粗糙程度，而是控制：

> 地图数值在空间中是零散跳变，还是大片连续。

大致含义：

* (H) 较小：局部变化快，地形更碎；
* (H) 较大：相邻区域更相似，地形更成片；
* 地形的平均粗糙度可以相同，但空间组织形式不同。

研究问题可以是：

> 不同 (H) 产生的环境结构，是否会诱导出不同的技能？

例如碎片化地形可能适合绕行，而大片高成本区域可能更容易产生修路技能。

---

### 4. 引入 Percolation 渗流模型

导师认为渗流模型与当前网格环境很像。

渗流模型主要控制的是：

> 随机地图中的可通行格子在什么条件下形成大型连通区域。

核心参数可以是 (p)：

* 以概率 (p) 生成可通行格子；
* 以概率 (1-p) 生成障碍；
* (p) 较低时，地图碎片化；
* (p) 较高时，形成大连通区；
* 临界区域附近会出现瓶颈、狭窄通道和多尺度结构。

在 EvoGrid 中可以分成两层：

* **连通层**：由 `OBSTACLE`、`DIG` 和参数 (p) 控制；
* **通行成本层**：由粗糙度、`ROAD` 和 `BUILD_ROAD` 控制。

因此：

* `DIG` 改变“能不能走”；
* `BUILD_ROAD` 改变“走起来有多贵”。

---

### 5. 研究 Agent 进化与环境进化的耦合

导师画图所表达的重点是：

> 环境变化推动 Agent 产生新能力，而 Agent 能力提升后，环境又继续升级。

可能形成阶梯式过程：

```text
环境阶段提升
→ Agent 遇到新问题
→ 形成新 Skill
→ Skill 通过验证
→ Agent 能力上升
→ 环境继续升级
```

需要比较不同环境课程：

* 环境和 Agent 能力匹配时，技能可能持续积累；
* 环境变化太快、太随机或没有合适学习压力时，技能可能停滞；
* 某些环境分布可能只能产生有限技能；
* 某些接近临界状态的环境可能持续诱导新技能。

---

### 6. Agent 能力应该用 Skill Set 衡量

导师图里的纵轴不应只使用 reward。

Reward 只表示某一次任务表现，而 Agent 的长期能力更应该表示为：

> 已形成、经过验证、可以复用和迁移的 Skill Set。

不能只计算技能数量，还需要考虑：

* 技能成功率；
* 是否能用于未见地图；
* 是否能跨地图尺寸迁移；
* 是否会在负场景错误触发；
* 是否可以和其他技能组合；
* 环境变化后是否仍有效。

因此可以区分：

* `Candidate Skill Set`：Agent 自己提出、尚未验证的技能；
* `Verified Skill Set`：在未见环境中验证通过的技能。

---

### 7. 必须区分 Prompt 和 Skill

这是研究中非常重要的边界。

**Prompt** 是外部临时提示：

> 告诉模型这一次应该注意什么、怎样思考或怎样行动。

**Skill** 是持久能力：

> 即使删除详细教学提示，仍可以在不同回合和未见环境中复用。

例如，Prompt 明确告诉模型：

```text
如果在运输路线上，未来使用次数超过回本次数，就修路。
```

模型照做，只能说明它理解了提示。

真正的 Skill 应该包括：

```text
识别重复运输需求
→ 找到高成本瓶颈
→ 估计未来复用
→ 比较回本次数
→ 决定是否修路
→ 后续复用并验证收益
```

而且应在没有详细修路教学 Prompt 的情况下仍然有效。

---

### 8. Skill 应该由 Agent 自己形成

导师提出的“自进化”不能只表现为：

> 研究者提前指定“修路技能”，Agent 只学习什么时候使用。

更强的目标应该是：

```text
Agent 反复执行任务
→ 发现重复失败或效率瓶颈
→ 自己提出候选 Skill
→ 在环境中尝试
→ 在未见地图验证
→ 有效则保存
→ 无效则修改或淘汰
```

目前系统仍然偏受控，因为人工提前规定了：

* `BUILD_ROAD` 是主要研究对象；
* 修路相关特征；
* `on_route`、`break_even_uses` 等证据；
* strong evidence 门槛；
* Prompt 如何使用这些证据；
* 正样本、混合样本和负样本场景。

这适合作为第一阶段机制验证，但还不能称为完整自主 Skill 演化。

## 二、我们对当前项目的定位

目前项目最准确的定位是：

> 一个环境塑造型技能学习的受控机制验证平台。

目前已经验证的部分包括：

* Agent 可以真实修改环境；
* 修路收益可以被记录和归因；
* 历史收益可以影响后续决策；
* 测试阶段关闭探索后，真实 LLM 仍可使用 learned evidence 修路；
* Agent 开始区分正、混合和负修路上下文。

但目前还没有完全实现：

* Skill 由 Agent 自主发现；
* Skill 自动形成、合并和淘汰；
* 环境根据 Agent 能力自动演化；
* Skill Set 持续增长；
* 小环境技能稳定迁移到大型开放世界；
* 环境改造稳定提升整体任务 reward。

## 三、关于地形表示的计划

### 1. 底层使用连续粗糙度

每个可通行格子保存：

[
r(x,y)\in[0,1]
]

而不是底层只有 `GROUND` 和 `ROUGH` 两类。

### 2. Agent 初期只观察离散档位

例如：

* `SMOOTH`
* `NORMAL`
* `ROUGH`
* `VERY_ROUGH`

这样既保留连续地形结构，也避免 Agent 输入一开始过于复杂。

### 3. 粗糙度不直接决定奖励

不采用：

[
R=-r
]

而采用更自然的因果链：

[
\text{粗糙度}
\rightarrow
\text{移动成本}
\rightarrow
\text{运输效率}
\rightarrow
\text{任务奖励}
]

例如：

[
C_{\text{move}}(r)=C_0+\alpha r^\gamma
]

修路将有效粗糙度降低，道路价值自然变成：

[
\text{未来使用次数}\times
\text{单次成本节省}
-\text{修路成本}
]

## 四、参数化地图生成计划

计划新增独立地图生成器，例如：

```text
FractalPercolationMapBuilder
```

其主要参数包括：

* `map_size`：地图尺寸；
* `p_open`：连通性参数；
* `H`：地形空间相关性；
* `roughness_strength`：粗糙度强度；
* `ore_distribution`：矿石均匀或聚集分布；
* `world_seed`：保证可复现。

推荐生成流程：

```text
1. 使用分形随机场生成连续地形值；
2. 用 H 控制地形聚集尺度；
3. 用阈值或 p 控制开放/障碍比例；
4. 计算连通分量和基地—矿点可达性；
5. 生成矿石分布；
6. 输出地图诊断信息。
```

第一阶段先只验证地图分布，不立即训练 Agent。

重点检查：

* 最大连通簇比例；
* 基地—矿点可达率；
* 最短路径和最低成本路径；
* 瓶颈数量；
* 高粗糙区域斑块大小；
* 不同参数是否真的产生不同地图结构。

## 五、大地图开放世界计划

不建议直接创建一张超大静态数组，而采用 chunk 分块生成：

```text
每个 chunk 为 32×32 或 64×64
Agent 接近边界时生成新区块
离开的区块保留环境改造记录
```

区块种子可以由：

```text
hash(world_seed, chunk_x, chunk_y)
```

确定。

需要长期保存：

* 已修道路；
* 已挖通道；
* 已采矿点；
* 区域访问记录；
* Skill 的使用和收益。

这样可以支持理论上持续扩展的世界。

## 六、环境—Agent 耦合实验计划

### 阶段一：固定参数环境族

研究不同参数对技能形成的影响：

[
(p,H,\text{size})
\rightarrow
\text{Skill emergence}
]

例如观察：

* 高连通环境是否主要形成导航 Skill；
* 粗糙运输环境是否形成修路 Skill；
* 临界连通环境是否形成瓶颈识别和 DIG Skill；
* 大地图是否形成区域记忆和层级规划 Skill。

### 阶段二：环境课程

先设计离散课程：

1. 小地图、高连通；
2. 重复运输粗糙走廊；
3. 正负修路上下文混合；
4. 接近渗流临界、有瓶颈；
5. 大地图、多矿区；
6. 环境发生扰动。

Agent 达到当前阶段能力要求后，环境才升级。

### 阶段三：自适应环境生成

环境根据 Agent 表现自动调整：

```text
成功率太高 → 提高难度
成功率太低 → 降低难度
处于学习边界 → 生成邻近新环境
```

研究环境是否能持续产生新的学习压力。

## 七、Skill 自主形成计划

计划引入通用 Skill 生命周期：

```text
Propose
→ Solve
→ Verify
→ Store / Revise / Delete
```

### Propose

Agent 分析成功和失败轨迹，自行提出候选 Skill：

```json
{
  "skill_name": "...",
  "problem_addressed": "...",
  "preconditions": {},
  "procedure": [],
  "expected_effect": "...",
  "source_episodes": []
}
```

不提前告诉它一定要生成“修路 Skill”。

### Solve

Agent 在新任务中尝试调用候选 Skill。

### Verify

独立验证器在未见地图中测试：

* 成功率；
* 正收益率；
* 负场景误触发率；
* 跨地图迁移；
* 是否优于无 Skill 基线。

### Store

通过验证后进入 `Verified Skill Set`。

### Revise / Delete

长期表现下降时：

* 修改适用条件；
* 合并重复 Skill；
* 降级为 Candidate；
* 删除失效 Skill。

## 八、需要设置的关键对照组

为了区分 Prompt、人工 Skill 和自主 Skill，建议设置：

| 组别                       | 含义                           |
| ------------------------ | ---------------------------- |
| No Skill                 | 只有基础任务提示，没有技能库               |
| Prompt-only              | 人工提供详细策略提示，但不保存 Skill        |
| Handcrafted Skill        | 使用当前人工定义的 RoadLearningModule |
| Self-proposed Skill      | Agent 自己从轨迹总结候选 Skill        |
| Self-proposed + Verified | 自生成 Skill 经独立验证后使用           |
| Prompt + Skill           | 详细提示和技能库联合，作为能力上限参考          |

最关键的比较是：

```text
Prompt-only
vs
Self-proposed + Verified
```

用来判断表现提升来自提示工程，还是来自持久技能。

## 九、Agent 能力图如何画

横轴：

* 环境课程阶段；
* 或参数化环境复杂度；
* 或环境演化步骤。

纵轴：

* Verified Skill Set 数量；
* 加权技能能力；
* 或技能覆盖矩阵。

每次阶梯上升表示：

> 一个新的候选 Skill 在未见环境中验证通过，并进入 Skill Set。

同时记录：

* 新 Skill 在什么环境中产生；
* 能否迁移到其他环境；
* 是否会遗忘；
* 是否在环境变化后失效；
* 是否能和旧 Skill 组合。

Reward 作为 Skill 验证信号之一，而不是 Agent 能力的唯一指标。

## 十、建议的实施顺序

### 当前阶段

继续保留现有 road-learning 系统，作为受控基线和机制验证平台。

### 下一步

1. 开发参数化地图生成器；
2. 独立验证 (p)、(H) 和地图尺寸是否真正控制环境结构；
3. 建立小地图到大地图的测试协议；
4. 定义 Candidate Skill 和 Verified Skill 数据结构；
5. 实现通用 `propose_skill()` 与 `verify_skill()`；
6. 比较 Prompt-only、人工 Skill 和自主 Skill；
7. 最后再加入自适应环境课程和大规模开放世界。

## 一句话总结

导师建议把项目从：

> 在受控地图中学习何时修路

推进为：

> 在由连通性、空间相关性和尺度共同控制的开放环境中，研究 Agent 如何自主发现、验证、积累和迁移 Skill，以及不同环境分布是否能够持续推动 Skill Set 演化。
