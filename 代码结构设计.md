# 代码结构设计：EvoGrid-Mine 实验系统

## 1. 设计目标

代码结构围绕第一次实验计划服务，优先保证：

1. 环境逻辑可控、可测试；
2. 训练、评估、可视化彼此解耦；
3. Full Shaping / No Shaping / Random / Greedy 可以复用同一套评估管线；
4. DeepSeek API 作为 LLM Agent 的决策和规划后端接入；
5. PPO、规则智能体和 LLM 智能体共享同一套环境、rollout、评估和可视化管线；
6. 支持 full-info 调试模式和 partial-observation 自进化模式；
7. 后续可以扩展到随机地图、多智能体、LLM 高层规划和 Craftax 迁移。

第一版代码不追求复杂工程框架，采用轻量 Python 包结构即可。

## 3. 核心模块职责

### 3.1 `evogrid/envs/`

环境层只负责世界状态、动作执行、奖励和指标记录，不关心训练算法。

| 文件 | 职责 |
| --- | --- |
| `evogrid_mine_env.py` | Gymnasium 风格环境，实现 `reset()`、`step()`、`render()` |
| `map_builder.py` | 创建固定地图和后续随机地图 |
| `map_state.py` | 保存地图矩阵、智能体位置、矿石、道路、挖掘痕迹 |
| `reward.py` | 集中定义奖励函数，避免奖励逻辑散落在环境里 |
| `metrics.py` | 统计道路使用率、改造次数、运输成本、轨迹热力图等 |

环境层需要支持两种观测模式：

```text
full_obs:
  用于调试和基线，可以暴露完整 grid、base_pos、ore_positions。

partial_obs:
  用于自进化实验，只暴露 local_view、当前状态和最近事件。
```

关键约束：

```text
partial_obs 不能直接暴露完整地图或矿石真值坐标。
Agent 如果知道矿点，必须来自历史局部观测和 memory。
```

环境对外暴露统一接口：

```python
env = EvoGridMineEnv(config)
obs, info = env.reset(seed=0)
obs, reward, terminated, truncated, info = env.step(action)
```

### 3.2 `evogrid/agents/`

智能体层负责根据观测选择动作。

| 文件 | 职责 |
| --- | --- |
| `base_agent.py` | 定义统一 Agent 接口，例如 `act(obs, info)`、`reset()` |
| `memory.py` | 保存 Agent 已观测到的地图信息、发现过的矿、失败动作和经验摘要 |
| `memory_route_planner.py` | 只基于 Agent 自身 memory 的低层导航模块，用于返航等路径执行 |
| `random_agent.py` | 随机策略 baseline |
| `greedy_agent.py` | 手写贪心策略 baseline |
| `partial_greedy_agent.py` | 只使用 partial_obs 和 memory 的非 LLM baseline |
| `ppo_agent.py` | Stable-Baselines3 PPO 封装 |
| `deepseek_agent.py` | 使用 DeepSeek API 进行动作决策或高层规划 |
| `self_evolution_agent.py` | partial_obs 下的 DeepSeek + memory + reflection 自进化 Agent |
| `hybrid_agent.py` | LLM 规划 + 规则/路径搜索执行，降低逐步调用成本 |

统一 Agent 接口建议如下：

```python
class BaseAgent:
    def reset(self, seed: int | None = None) -> None:
        ...

    def act(self, obs, info: dict) -> int:
        ...
```

其中 `DeepSeekAgent` 是 DeepSeek API 的主要使用者。它不应该直接散落在训练脚本里调用 API，而是通过 `agents/deepseek_agent.py` 统一封装：

```text
observation + map summary + task state
        ↓
DeepSeekAgent
        ↓
evogrid.llm.DeepSeekClient
        ↓
structured JSON decision
        ↓
action id / subgoal / route plan
```

第一轮实验可以同时保留两条线：

1. PPO 主实验：验证“可学习智能体是否能学会环境塑造”；
2. DeepSeek Agent 对照实验：验证“大模型规划能力是否能直接产生合理的修路/挖通道策略”；
3. 自进化实验：验证 Agent 是否能在 partial_obs 下通过探索、记忆和反思逐渐形成环境塑造倾向。

### 3.3 `evogrid/llm/`

DeepSeek 相关底层能力集中放在 `llm/`。这一层不直接扮演智能体，只提供 API 调用、prompt、结构化输出解析和规划工具；真正的智能体封装放在 `evogrid/agents/deepseek_agent.py`。

| 文件 | 职责 |
| --- | --- |
| `deepseek_client.py` | DeepSeek API 客户端封装 |
| `prompts.py` | 动作决策、阶段性规划、地图摘要、实验分析 prompt |
| `reflection.py` | 根据 episode 结果生成下一轮策略记忆和反思 |
| `schemas.py` | LLM 输入输出结构，例如 JSON schema 或 dataclass |
| `parser.py` | 从 DeepSeek 文本响应中提取并校验 JSON |
| `planner.py` | 将 DeepSeek 输出转换为高层规划建议 |

DeepSeek 在本项目中有三种使用模式：

1. **Step-level Agent**：每一步根据当前观测直接输出动作；
2. **Planner Agent**：每隔 N 步输出一个高层子目标，例如“挖开坐标 x,y 的障碍”或“沿当前路线修路”，再由规则执行器落地；
3. **Analysis Assistant**：读取实验指标和图表摘要，生成实验报告与失败诊断。

第一次实验更推荐使用 **Planner Agent**，而不是每一步都调用 DeepSeek。原因是每步调用成本高、延迟大，而且动作空间很低层。Planner Agent 更符合 LLM 的优势：看地图、判断瓶颈、提出建设目标。

自进化实验中，DeepSeek prompt 不能直接读取完整地图或矿点真值。它只能读取：

```text
partial_obs local_view
Agent memory 摘要
recent_events
上一轮 reflection
当前指标摘要
```

这保证策略变化来自经验，而不是环境把答案直接告诉 Agent。

`AgentMemory` 建议保存：

```json

  "seen_ore_locations": [],
  "seen_obstacles": [],
  "seen_rough_tiles": [],
  "visited_counts": {},
  "built_roads": [],
  "dug_cells": [],
  "failed_actions": [],
  "successful_routes": [],
  "reflections": []

```

`reflection.py` 输出建议为：

```json

  "lessons": ["..."],
  "avoid": ["..."],
  "try_next": ["..."],
  "exploration_bias": "...",
  "environment_shaping_hypothesis": "..."

```

注意：`memory.py` 只允许从 observation 和 action result 更新，不能直接读取环境内部真值。

### 3.4 `evogrid/training/`

训练层负责运行 PPO 和保存训练过程。

| 文件 | 职责 |
| --- | --- |
| `train_ppo.py` | 单组 PPO 训练入口 |
| `callbacks.py` | 定期评估、保存模型、记录曲线 |
| `rollout.py` | 运行训练后策略并保存轨迹 |

训练入口应支持：

```text
Full Shaping
No Shaping
Road Only

High Cost Shaping
```

第一次实验先实现 Full Shaping 和 No Shaping。

### 3.5 `evogrid/evaluation/`

评估层负责统一比较不同实验组。

| 文件 | 职责 |
| --- | --- |
| `evaluate_policy.py` | 对单个模型或 baseline 运行评估 |
| `compare_groups.py` | 汇总多个实验组的均值、方差、显著性结果 |
| `summary.py` | 生成 CSV / JSON 摘要 |

输出建议保存为：

```text

outputs/metrics/no_shaping_seed0_eval.csv
outputs/metrics/first_experiment_summary.csv
```

### 3.6 `evogrid/visualization/`

可视化层只消费日志和地图状态，不重新运行训练逻辑。

| 文件 | 职责 |
| --- | --- |
| `render_map.py` | 渲染单帧地图和 before/after 对比 |
| `plot_curves.py` | 画奖励、交付量、运输成本曲线 |
| `plot_heatmaps.py` | 画轨迹热力图和改造热力图 |

## 4. DeepSeek API 接入设计

### 4.1 环境变量

`.env.example` 建议包含：

```text
DEEPSEEK_API_KEY=your_deepseek_api_key_here
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
DEEPSEEK_TIMEOUT=30
```

真实 `.env` 不提交到 Git。

### 4.2 客户端接口

`deepseek_client.py` 建议提供一个极薄封装：

```python
class DeepSeekClient:
    def __init__(self, api_key: str, base_url: str, model: str, timeout: int = 30):
        ...

    def chat(self, messages: list[dict], temperature: float = 0.2) -> str:
        ...

    def chat_json(self, messages: list[dict], temperature: float = 0.0) -> dict:
        ...
```

设计原则：

- API key 只从环境变量或配置文件读取；
- 不在日志里打印 API key；
- LLM 返回结果必须经过 JSON 解析和字段校验；
- DeepSeek 调用失败时不影响 PPO 训练主流程，DeepSeek Agent 使用 fallback 继续运行；
- LLM 决策 trace 保存到 `outputs/runs/.../llm_trace.jsonl`；
- LLM 分析报告保存到 `outputs/llm_reports/`，便于复查。

### 4.3 DeepSeek Agent 决策接口

DeepSeek Agent 的核心输出必须是结构化 JSON，不能依赖自由文本解释来执行动作。

Step-level 决策输出建议：

```json
{
  "mode": "action",
  "action": "BUILD_ROAD",
  "action_id": 6,
  "reason": "The current cell lies on the repeated path between base and ore.",
  "confidence": 0.72
}
```

Planner-level 决策输出建议：

```json
{
  "mode": "plan",
  "subgoal": "build_transport_corridor",
  "target_cells": [[3, 2], [4, 2], [5, 2]],
  "preferred_actions": ["BUILD_ROAD", "MOVE_RIGHT"],
  "stop_condition": "agent_reaches_ore_or_corridor_complete",
  "reason": "A corridor between base and ore can reduce repeated transport cost."
}
```

`DeepSeekAgent` 负责把这些结构化结果转换为环境动作：

```text
DeepSeek JSON
    ↓
parser.py 校验字段
    ↓
planner.py 转成 subgoal 或 primitive action
    ↓
deepseek_agent.py 输出 action_id
    ↓
env.step(action_id)
```

如果 DeepSeek 返回非法 JSON、动作越界或 API 调用失败，`DeepSeekAgent` 应回退到安全策略：

```text
优先执行当前 plan 的下一步
否则使用 GreedyAgent
否则 NOOP
```

正式自进化实验中，还需要增加动作合法性过滤：

```text
不在基地且未携带矿时，DROPOFF 无效，应过滤或回退；
不在矿点或矿旁边时，MINE 无效，应过滤或回退；
当前格不可修路时，BUILD_ROAD 无效，应过滤或回退；
附近无障碍时，DIG 无效，应过滤或回退。
```

这属于通用安全执行能力，不属于硬编码修路策略。

### 4.4 第一次实验中的 DeepSeek 用法

第一次实验中 DeepSeek 应作为一个正式对照智能体加入，而不是只做分析工具。

建议增加实验组：

```text
DeepSeek Step Agent
DeepSeek Planner Agent
Hybrid DeepSeek + Greedy Agent
```

优先实现顺序：

1. **Hybrid DeepSeek + Greedy Agent**：DeepSeek 只负责给出建设子目标，Greedy/路径搜索负责执行；
2. **DeepSeek Planner Agent**：每隔 10 到 25 步重新规划一次；
3. **DeepSeek Step Agent**：每一步调用 DeepSeek，主要用于小地图调试，不建议作为大规模训练组。

这样可以回答两个问题：

1. RL 智能体是否能通过训练学会环境塑造；
2. DeepSeek 智能体是否能凭借显式地图理解直接提出环境塑造策略。

但这仍然不是严格自进化。严格自进化版本需要：

```text
partial_obs
Agent memory
episode reflection
跨 episode 策略记忆
禁止矿点真值泄露
```

### 4.5 DeepSeek 实验分析用法

实验结束后，DeepSeek 仍可以作为分析助手：

```text
metrics csv/json
      ↓
DeepSeek 分析 prompt
      ↓
自然语言实验报告
      ↓
outputs/llm_reports/first_experiment_analysis.md
```

输入给 DeepSeek 的内容应是结构化摘要，而不是完整训练日志：

```json
{
  "experiment": "first_experiment",
  "groups": {
    "full_shaping": {
      "mean_reward": 0.0,
      "mean_ore_delivered": 0.0,
      "mean_road_usage_rate": 0.0
    },
    "no_shaping": {
      "mean_reward": 0.0,
      "mean_ore_delivered": 0.0
    }
  },
  "main_question": "Does environment shaping improve long-term resource collection?"
}
```

### 4.6 后续 DeepSeek 扩展

第二阶段可以扩展为：

1. partial-observation 自进化 Agent；
2. 跨 episode memory 和 reflection；
3. 多智能体 LLM 角色分工：builder、miner、carrier；
4. LLM 高层规划 + RL 低层控制；
5. LLM 失败诊断：根据训练曲线建议调整奖励、地图和 episode 长度；
6. LLM 实验报告生成：自动整理图表解读和下一步建议；
7. LLM 课程学习设计：根据当前失败模式生成下一张训练地图。

## 6. 数据流

### 6.1 训练数据流

```text
config
  ↓
EvoGridMineEnv
  ↓
PPO / Random / Greedy / DeepSeekAgent / HybridAgent
  ↓
episode info
  ↓
outputs/runs/
  ↓
evaluation summary
  ↓
outputs/metrics/
```

### 6.2 DeepSeek Agent 决策数据流

```text
EvoGridMineEnv observation
  ↓
observation summarizer
  ↓
DeepSeekAgent
  ↓
DeepSeekClient
  ↓
JSON action / JSON plan
  ↓
parser + planner
  ↓
env.step(action_id)
  ↓
episode metrics + LLM trace
```

自进化版本的数据流为：

```text
partial_obs local_view
  ↓
memory.update_from_observation()
  ↓
DeepSeek prompt = local_view + memory summary + reflection
  ↓
JSON action / plan
  ↓
action legality filter
  ↓
env.step(action_id)
  ↓
memory.update_from_result()
  ↓
episode metrics
  ↓
reflection.generate()
  ↓
next episode prompt
```

DeepSeek Agent 的每次调用建议额外保存 trace：

```text
outputs/runs/deepseek_planner_seed0/llm_trace.jsonl
```

trace 至少包含：

```text
step
observation_summary
prompt_hash
raw_response
parsed_decision
chosen_action
fallback_used
```

### 6.3 可视化数据流

```text
saved rollout
  ↓
map states + visited mask + changed mask
  ↓
render_map / plot_heatmaps
  ↓
outputs/figures/ and outputs/videos/
```

### 6.4 DeepSeek 分析数据流

  ↓

  ↓
outputs/llm_reports/first_experiment_analysis.md

## 9. 测试设计

第一版测试不需要覆盖训练效果，但必须覆盖环境正确性。

| 测试文件 | 测试内容 |
| --- | --- |
| `test_env_step.py` | 移动、撞墙、越界、终止条件 |
| `test_map_builder.py` | 固定地图可达性、元素位置正确 |
| `test_rewards.py` | 交付、移动、挖掘、修路、无效动作奖励 |
| `test_metrics.py` | 道路使用率、改造格子数、轨迹记录 |
| `test_deepseek_client.py` | 使用 mock 测试 JSON 解析和错误处理 |
| `test_deepseek_agent.py` | 使用 mock DeepSeek 响应测试动作映射、fallback 和 trace 记录 |

最重要的 smoke test：

```text
随机智能体运行 10 个 episode 不报错；
Greedy 智能体能至少完成 1 次资源交付；
Full Shaping 中 BUILD_ROAD 和 DIG 能真实改变地图；
No Shaping 中 BUILD_ROAD 和 DIG 不会改变地图；
mock DeepSeek Agent 能把 JSON action 转成正确 action_id；
mock DeepSeek Planner 能执行一个简单修路子目标。
```

## 10. 关键设计约束

1. 环境不直接依赖 DeepSeek；
2. DeepSeek 只通过 Agent 层接入环境；
3. PPO 训练不直接依赖 DeepSeek；
4. DeepSeek 失败不应导致整体实验失败，Agent 必须有 fallback；
5. 地图状态、指标记录和 LLM trace 必须可序列化；
6. 所有实验配置从 YAML 读取；
7. 所有输出进入 `outputs/`；
8. API key 不进入代码、不进入日志、不进入 Git；
9. 自进化实验中，Agent 不得直接读取矿点真值或完整地图；
10. 不硬编码“修路/挖掘”策略，只实现通用观测、记忆、反思和合法性过滤。
