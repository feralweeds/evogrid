# 第一次实验计划：EvoGrid-Mine 最小机制验证

## 1. 实验目标

第一次实验只验证一个核心问题：

> 在同一张 2D 采矿地图上，允许环境改造的智能体，是否能通过修路或挖通道获得高于禁止环境改造智能体的长期收益？

本次实验不追求完整论文级结论，而是作为机制验证。成功标准是能够看到：

1. Full Shaping 的资源交付量和累计奖励高于 No Shaping；
2. 智能体会主动执行 `BUILD_ROAD` 或 `DIG`；
3. 改造格子集中在基地与矿区之间，而不是随机散布；
4. 被建造的道路或挖开的通道在后续运输中被反复使用。

### 1.1 当前研究方向修正

当前已实现的第一版环境是 **full-info 调试版本**：Agent 可以在代码层拿到完整地图、基地位置和矿石位置。这个版本适合验证环境、奖励、指标、DeepSeek API、图表和评估管线，但不适合作为“自进化”主结论。

如果 Agent 一开始就知道矿石全局坐标，任务会退化成：

```text
已知目标点路径规划
```

而不是：

```text
未知环境中的探索、记忆、自我反思和环境塑造
```

因此后续正式实验需要切换到 **partial-observation 自进化版本**：

```text
局部观测 -> 探索发现 -> 记忆积累 -> episode 后反思 -> 下一轮策略调整
```

full-info 结果只能作为诊断基线，不应被解释为自进化证据。

## 2. 实验范围

### 本次包含

- 自研极简 Gymnasium 风格环境；
- 单智能体；
- 固定 32 x 32 地图；
- 1 个基地；
- 1 个主要矿区；
- 障碍物、高成本地形、道路；
- `MOVE`、`MINE`、`DIG`、`BUILD_ROAD`、`DROPOFF`、`NOOP`；
- PPO 训练；
- Full Shaping vs No Shaping 对照；
- Random 与 Greedy 作为非学习基线；
- 训练曲线、地图前后对比、轨迹热力图、改造热力图。

### 本次不包含

- 多智能体；
- 竞价或任务分配；
- 仓库、中转点、桥梁、塌方；
- 跨 episode 持久化世界；
- Craftax-Classic 迁移；
- 大规模随机地图泛化。

## 3. 环境配置

| 项目 | 设置 |
| --- | --- |
| 环境名称 | EvoGrid-Mine |
| 地图大小 | 32 x 32 |
| 智能体数量 | 1 |
| 基地数量 | 1 |
| 矿区数量 | 1 |
| 每轮最大步数 | 500 |
| 观测形式 | 调试阶段可用 full_obs；正式自进化实验使用 partial_obs |
| 地图类型 | 固定地图 |
| 随机种子 | 5 个种子：0、1、2、3、4 |

固定地图应刻意制造“建设有价值”的结构：

- 基地与矿区之间存在可通行绕路；
- 中间有障碍墙或高成本地形；
- 直接挖通或修路后，运输路线明显缩短；
- 如果不改造环境，智能体仍能完成任务，但效率较低。

### 3.1 观测设置

#### full_obs：调试模式

full_obs 可以包含：

```text
完整 grid
agent_pos
base_pos
ore_positions
has_ore
step
ore_delivered
```

它用于：

```text
环境调试
Greedy/PPO baseline
指标和可视化验证
DeepSeek API smoke test
```

full_obs 不作为自进化主实验设定。

#### partial_obs：正式自进化模式

partial_obs 不直接提供：

```text
完整 grid
ore_positions
全局障碍位置
全局粗糙地位置
全局最短路径
```

partial_obs 只提供：

```text
local_view
agent_pos 或相对坐标
base_pos 或 home_direction
has_ore
step
ore_delivered
recent_events
```

如果 Agent 知道某个矿点位置，必须是因为它曾经在局部视野里发现过，并把它写入 memory。

## 4. 动作空间

```text
0 = MOVE_UP
1 = MOVE_DOWN
2 = MOVE_LEFT
3 = MOVE_RIGHT
4 = MINE
5 = DIG
6 = BUILD_ROAD
7 = DROPOFF
8 = NOOP
```

No Shaping 组中，`DIG` 和 `BUILD_ROAD` 被禁用。禁用方式建议保持动作空间大小不变，但执行后视为无效动作并给予无效动作惩罚，这样便于 PPO 网络结构复用。

## 5. 奖励函数

基础奖励：

```text
交付 1 单位矿石：+10
普通移动：-0.01
高成本地形移动：-0.05
DIG：-0.2
BUILD_ROAD：-0.1
无效动作：-0.05
```

关键原则：

- 不直接奖励修路；
- 不直接奖励挖通道；
- 环境改造只能通过降低后续运输成本间接提升收益。

## 6. 实验组

### A. Full Shaping

允许完整动作空间，包括：

```text
DIG
BUILD_ROAD
```

这是主实验组。

### B. No Shaping

禁止环境改造动作，只允许：

```text
MOVE
MINE
DROPOFF
NOOP
```

这是关键对照组。

### C. Random Agent

随机选择动作，用作下界。

### D. Greedy Agent

手写贪心策略：

- 若未携带矿石，走向最近矿区；
- 若携带矿石，走向基地；
- 不主动修路；
- 不主动挖通道，除非后续实现中必须挖掘才能保持可达。

Greedy 用于判断 PPO 是否只是学到短视采矿，还是真正超过短视策略。

### E. DeepSeek Planner Agent

DeepSeek 不逐步控制每个低层动作，而是每隔固定步数读取观测摘要、当前任务状态和自身记忆，输出高层计划，例如：

```text
挖开某段障碍
在基地到矿区之间修一段路
先完成道路走廊，再开始高频运输
```

环境执行层再把计划转成具体动作。正式自进化实验中，DeepSeek 不能直接读取完整地图或矿石真值坐标，只能读取局部视野和自身探索记忆。

### F. Hybrid DeepSeek + Greedy Agent

DeepSeek 负责阶段性规划，Greedy 或路径搜索负责低层移动和采矿。

这是第一轮最推荐的 LLM Agent 组，因为它能发挥 DeepSeek 的规划优势，同时避免每一步都调用 API 带来的延迟和成本。

## 7. 训练设置

优先使用：

```text
PPO + MLP policy
```

建议训练预算：

```text
每组 5 个随机种子
每个种子 200k 到 500k environment steps
每 10k steps 做一次评估
每次评估运行 20 个 episode
```

如果 200k steps 内完全看不到学习迹象，先不要盲目加大训练量，优先检查地图、奖励和动作有效性。

DeepSeek Agent 不需要训练，但需要固定评估预算：

```text
每组 20 到 50 个 episode
Planner 每 10 到 25 步调用一次 DeepSeek
保存每次 LLM 调用 trace
API 失败时 fallback 到 Greedy 或 NOOP
```

## 8. 记录指标

每个 episode 至少记录：

| 指标 | 含义 |
| --- | --- |
| `episode_reward` | 累计奖励 |
| `ore_delivered` | 成功交付矿石数量 |
| `num_dig` | 挖掘次数 |
| `num_build_road` | 修路次数 |
| `invalid_actions` | 无效动作次数 |
| `road_cells_built` | 建成道路格子数 |
| `dug_cells` | 挖开障碍格子数 |
| `road_usage_rate` | 被走过的道路格子数 / 建成道路格子数 |
| `transport_steps_per_ore` | 平均每单位矿石运输步数 |
| `early_shaping_ratio` | 前 100 步内改造动作占比 |
| `late_delivery_rate` | 后 250 步单位时间交付量 |
| `llm_calls` | DeepSeek 调用次数，仅 LLM Agent 组记录 |
| `llm_fallbacks` | DeepSeek 失败或非法输出后的 fallback 次数 |
| `llm_plan_success_rate` | LLM 子目标被完成的比例 |
| `discovered_ore_count` | 通过局部观测发现的矿点数量 |
| `known_ore_count` | Agent memory 中记住的矿点数量 |
| `map_coverage_rate` | 已观测地图格子占比 |
| `memory_updates` | episode 内记忆更新次数 |
| `reflection_updates` | episode 后反思更新次数 |

第一轮最关键的三个指标：

1. `ore_delivered`
2. `episode_reward`
3. `road_usage_rate`

## 9. 可视化产物

第一次实验结束后至少生成：

1. 奖励曲线：Full Shaping vs No Shaping vs Random vs Greedy；
2. 资源交付曲线；
3. 平均运输成本曲线；
4. 道路使用率曲线；
5. Full Shaping 的 episode 开始/结束地图对比；
6. Full Shaping 的轨迹热力图；
7. Full Shaping 的改造热力图；
8. 一段训练后策略 rollout 动图或视频。

## 10. 成功标准

第一次实验可以判定为成功，如果满足以下条件中的大部分：

1. Full Shaping 平均 `ore_delivered` 高于 No Shaping；
2. Full Shaping 平均 `episode_reward` 高于 No Shaping；
3. Full Shaping 中 `num_build_road + num_dig > 0`，且不是偶然个例；
4. `road_usage_rate` 明显高于随机水平；
5. 改造热力图集中在基地与矿区之间；
6. 训练后期的 `transport_steps_per_ore` 低于训练早期。

如果只有奖励提升，但没有功能性改造证据，则不能认为核心假设已经被验证。

自进化版本还需要额外满足：

1. Agent 初始不知道矿点全局位置；
2. 矿点信息来自局部观测发现，而不是环境真值注入；
3. Agent memory 随 episode 增长而更新；
4. Reflection 会影响下一轮 prompt 或行为倾向；
5. 修路、挖掘等环境塑造行为不能来自硬编码规则；
6. 若出现改造行为，需要能追溯到 memory/reflection 中的经验假设。

## 11. 失败诊断顺序

如果 Full Shaping 没有优势，按以下顺序排查：

1. 地图是否太简单：不修路也能高效采矿；
2. 地图是否太难：随机探索几乎找不到矿区或基地；
3. 修路收益是否太弱：道路移动成本差异不明显；
4. 修路或挖掘成本是否太高；
5. episode 是否太短，建设收益来不及回收；
6. 无效动作比例是否过高；
7. PPO 是否没有足够探索；
8. 观测中是否明确包含道路、障碍、矿区、基地和智能体位置。

如果自进化版本没有出现环境塑造，优先排查：

1. partial_obs 是否过窄，导致几乎发现不了矿；
2. memory 是否正确记录已发现矿点、障碍、粗糙地和路线；
3. reflection 是否真的进入下一轮 prompt；
4. prompt 是否只要求即时动作，缺少长期假设；
5. episode 是否太短，探索和改造收益来不及体现；
6. 指标是否能衡量探索覆盖率和记忆变化；
7. 是否不小心把矿石真值坐标泄露给了 DeepSeek。

对应调整优先级：

1. 先调整地图结构，让“改造有明显价值”；
2. 再调整 `BUILD_ROAD` 和 `DIG` 成本；
3. 再把 episode 从 500 增加到 1000；
4. 最后再考虑课程学习或换算法。

## 12. 执行顺序

### 第 1 步：实现最小环境

- 定义地图元素；
- 实现 reset 和 step；
- 实现动作合法性；
- 实现奖励；
- 记录 changed mask 和 visited mask；
- 写一个随机策略 smoke test。

### 第 2 步：固定地图调试

- 确认基地到矿区可达；
- 确认挖通道可以缩短路径；
- 确认修路可以降低运输成本；
- 确认 No Shaping 仍可完成任务但效率较低。

### 第 3 步：实现 baseline

- Random Agent；
- Greedy Agent；
- 统一 Agent 接口；
- 统一 rollout 和 metrics 保存。

### 第 4 步：接入 DeepSeek Agent

- 实现 DeepSeek API client；
- 实现局部观测和记忆摘要 prompt；
- 实现 JSON action / plan schema；
- 实现 DeepSeek Planner Agent；
- 实现 Hybrid DeepSeek + Greedy Agent；
- 用 mock 响应做 smoke test；
- 再用真实 DeepSeek API 跑 20 个 episode。

### 第 4.5 步：接入自进化记忆

- 实现 partial_obs；
- 实现 Agent memory；
- 记录局部视野中首次发现的矿、障碍、粗糙地和道路；
- episode 后根据指标和轨迹生成 reflection；
- 下一轮 prompt 读取 memory 和 reflection；
- 验证 memory 只来自已观测信息，不读取环境真值。

### 第 5 步：实现 PPO 训练脚本

- Full Shaping PPO；
- No Shaping PPO；
- 定期评估；
- 保存模型和训练日志。

### 第 6 步：跑主实验

- Full Shaping，5 个种子；
- No Shaping，5 个种子；
- DeepSeek Planner，20 到 50 个 episode；
- Hybrid DeepSeek + Greedy，20 到 50 个 episode；
- 每 10k steps 评估一次；
- 保存模型、日志和图表数据。

### 第 7 步：生成图表和定性证据

- 训练曲线；
- before/after 地图；
- 轨迹热力图；
- 改造热力图；
- LLM plan trace 摘要；
- rollout 视频或 GIF。

### 第 8 步：写第一次实验结论

结论不只报告“奖励是否更高”，还要回答：

- 智能体有没有主动改造环境？
- 改造是否集中在关键路径？
- 改造是否被后续使用？
- 性能优势是否来自更低运输成本？
- DeepSeek Agent 的规划是否比 Greedy 更接近有效环境塑造？

## 13. 预期结论模板

如果实验成功，可以形成如下结论：

> 在固定 32 x 32 EvoGrid-Mine 环境中，Full Shaping 智能体相较 No Shaping 获得更高资源交付量和累计奖励。训练后的策略会在基地与矿区之间主动修路或挖通道，这些改造格子在后续运输中被高频使用，并伴随平均运输成本下降。这说明智能体初步学会了通过短期环境改造换取长期任务收益。

如果实验失败，则结论应聚焦失败原因，而不是强行解释：

> 在当前地图和奖励设置下，未观察到稳定的环境塑造策略。下一步应优先检查地图是否能体现建设收益，并调整道路收益、改造成本和 episode 长度。
