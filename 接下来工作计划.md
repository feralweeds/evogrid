## 3. 接下来最重要的目标

当前已经完成了第一次实验闭环和真实 DeepSeek 接入，但这暴露了一个重要研究问题：

> 如果一开始就把矿石全局坐标、基地坐标和地图结构直接给 Agent，任务会很容易退化成路径规划，而不是自进化。

因此后续目标需要从“让 DeepSeek 会调用 API 并参与对比”调整为：

> 构建部分可观测环境，让 Agent 通过探索、记忆、反思逐渐发现矿、路线和环境改造价值。

第一阶段的 full-info 版本仍然有价值，但它应定位为调试基线，用来验证环境、动作、奖励、指标、图表、DeepSeek API 和评估管线是否稳定。

接下来不是马上跑大实验，而是先完成第一次实验闭环：

> 一键运行 Full Shaping、No Shaping、Random、Greedy，并保存统一指标，最后生成可以看的对比结果。

第一阶段要证明的不是“模型一定很强”，而是：

1. 训练脚本能稳定跑；
2. 每组实验结果能保存；
3. 指标字段齐全；
4. 能比较 Full Shaping 和 No Shaping；
5. 能看到是否出现修路、挖掘、道路使用等环境塑造行为。

从自进化研究角度看，正式实验还需要进一步满足：

1. Agent 不能一开始知道矿石全局坐标；
2. Agent 不能直接看到完整地图；
3. Agent 只能通过局部观测发现资源、障碍、粗糙地形和路线；
4. Agent 可以保存自己探索得到的记忆；
5. Agent 可以基于上一轮结果生成下一轮策略反思；
6. 不能把“遇到粗糙地就修路”“沿矿石路径铺路”这类答案硬编码进策略。

## 任务 8：部分可观测环境与自进化记忆（下一步）

这一任务是研究方向修正后的下一步。

目标不是教 Agent 修路，而是让 Agent 在更合理的信息条件下自发形成策略：

```text
局部观测 -> 探索 -> 发现矿/障碍/粗糙地 -> 记录记忆 -> episode 后反思 -> 下一轮调整策略
```

### 8.1 为什么要做

当前 `obs` 里包含：

```text
完整 grid
agent_pos
base_pos
ore_positions
has_ore
step
ore_delivered
```

DeepSeek prompt 里虽然没有直接放完整 grid，但仍然给了：

```text
ore_positions
base_pos
agent_pos
local_view
tile_counts
```

这导致 DeepSeek 和 Greedy 都可以把问题简化成“从基地到矿点再回来”的导航问题。

自进化实验中，Agent 不应该一开始知道矿点。矿点应该通过探索被发现，并进入记忆。

### 8.2 新增 observation mode

建议增加：

```text
full_obs     调试模式，保留当前全信息观察
partial_obs  正式自进化实验模式
```

`full_obs` 用于环境调试和对照。

`partial_obs` 中不直接提供：

```text
完整 grid
ore_positions
全局障碍位置
全局粗糙地位置
```

`partial_obs` 只提供：

```text
local_view
agent_pos 或相对位置
base_pos 或 home_direction
has_ore
step
ore_delivered
recent_events
```

### 8.3 新增 Agent memory

Agent 可以保存自己探索得到的信息：

```text
seen_tiles
seen_ore_locations
seen_obstacles
seen_rough_tiles
visited_counts
built_roads
dug_cells
failed_actions
successful_routes
```

关键原则：

```text
memory 只能来自 Agent 已经观测到的局部视野和动作结果
不能从环境真值直接灌入
```

### 8.4 DeepSeek prompt 修正

DeepSeek 正式实验 prompt 只能包含：

```text
当前 local_view
当前任务状态
Agent memory 摘要
最近 N 步事件
上一轮 reflection
可选动作
```

不再直接包含：

```text
ore_positions 真值
完整地图
```

## 8. 当前关键问题：自学习修路

最新 `self_evolution_route_planner_10ep` 结果说明，当前系统已经基本跑通：

```text
局部观察 -> 发现矿 -> 挖矿 -> 基于记忆地图返航 -> 交付
```

其中 `MemoryMapRoutePlanner` 解决的是低层导航问题。它只基于 Agent 自己观察和记住的地图规划返航路线，不读取隐藏的完整环境真值。因此，当前主要问题已经不是“Agent 不会回基地”，而是：

```text
Agent 如何从自己的交互经验中学会判断什么时候修路值得？
```

这一步不能通过硬编码规则完成。不能直接写：

```text
如果 rough 就 BUILD_ROAD
如果在基地到矿之间就修路
如果当前路径会重复经过就自动修路
```

这些规则会把研究问题变成“人工规则是否有效”，而不是“Agent 是否能自学习环境塑造策略”。

### 8.1 为什么当前 BUILD_ROAD 为 0

当前 `BUILD_ROAD = 0` 不一定是 bug，主要有三个原因：

1. 返航时 `RoutePlanner` 接管太强。它负责“怎么走”，但不会主动判断“当前格子是否值得改造成道路”。
2. 系统缺少修路收益的信用分配。Agent 还没有看到一次修路行为在后续 episode 中到底赚了还是亏了。
3. 修路本身不是总是合理。当前奖励下，普通 ground 修路大约需要未来经过 10 次才回本，rough terrain 修路大约经过 2 次就可能回本。

因此 Agent 不应该学到“看到地面就修路”，而应该逐渐学到：

```text
常走的 rough corridor 值得修；
普通 ground 只有在高频运输路径上才值得修；
修完之后很少再经过的路是浪费。
```

### 8.2 必须补上的信用分配

下一步需要加入 `RoadCreditTracker`，记录每条由 Agent 建造的道路：

```text
road_position
original_tile_type
original_move_cost
road_move_cost
build_step
future_usage_count
saved_cost
net_payoff
```

核心学习信号定义为：

```text
road_payoff =
  future_usage_count * (original_tile_move_cost - road_move_cost)
  - build_road_cost
```

例如 rough terrain 上修路：

```text
build_road_cost = 0.1
rough_move_cost = 0.05
road_move_cost = 0.0
future_usage_count = 3

road_payoff = 3 * (0.05 - 0.0) - 0.1 = +0.05
```

这说明这次修路是有收益的。如果修完后没有再经过：

```text
road_payoff = 0 - 0.1 = -0.1
```

这说明这次修路是浪费。

实现时一定要记录 `original_tile_type`。因为一旦执行 `BUILD_ROAD`，当前格子就会变成 road，如果不保存原始地形，就无法判断它原来是 ground 还是 rough，也就无法计算真实节省。

### 8.3 不能泄露未来

`RoadLearningModule` 可以使用过去 episode 的统计结果，例如：

```text
过去 rough corridor 修路通常收益为正
过去普通 ground 随机修路通常收益为负
```

但在当前时刻不能使用：

```text
这条路未来真实会被走几次
未来完整路线
隐藏矿点
完整地图
全局最优修路点
```

当前时刻只能使用 Agent 已知或可估计的信息：

```text
当前 tile 类型
当前格是否可 BUILD_ROAD
当前格过去访问过几次
当前格是否在当前记忆路径上
当前格是否是 rough terrain
剩余步数
是否 carrying ore
过去相似修路样本的平均 payoff
```

### 8.4 冷启动问题

现在没有修过路，就没有 road payoff 数据。因此需要一个冷启动机制，否则 `RoadLearningModule` 永远没有样本。

允许的冷启动方式：

```text
少量探索性修路
epsilon-greedy build road
LLM 偶尔尝试高潜力候选 road
先用 rule-road oracle 生成 sanity check 数据
curriculum 中构造 rough corridor
```

这里的边界是：冷启动不是告诉 Agent 哪里必须修，而是允许它在有潜在价值的位置尝试修路，然后从结果中学习。

### 8.5 推荐模块边界

不要加入一个自动执行的规则型修路器。推荐边界是：

```text
RoutePlanner:
只负责回答怎么走。

RoadCreditTracker:
记录每条路的原始地形、建造成本、后续使用次数、节省成本和净收益。

RoadLearningModule:
根据 Agent 自己过去的 road_payoff 经验，估计当前格子修路是否值得。

ShapingOpportunity:
在 RoutePlanner 给出移动动作前，把当前是否存在修路机会的信息交给 LLM / policy。

LLM / high-level policy:
根据候选信息，在 BUILD_ROAD 和 route action 之间做最终选择。
```

最重要的约束：

```text
系统不能自动执行 BUILD_ROAD。
```

系统可以提供“当前格可能值得修路”的候选信息，但最终是否修路必须由 Agent / policy 决定。

任务 8.4：RoadLearningModule
从历史 road_payoff 学习修路倾向。

`RoadLearningModule` 的定位是价值估计器，不是动作执行器。它从 `AgentMemory.road_credit_records` 中读取历史样本，按原始地形聚合：

```text

mean_payoff
positive_rate
mean_usage
confidence
```

```json

  "positive_rate": 0.5,
  "mean_usage": 10.5,
  "evidence_count": 2,

    "sample_count": 9,
    "positive_count": 9,
    "mean_payoff": 1.22,
    "positive_rate": 1.0,
    "mean_usage": 26.5
  },
  "uses_hidden_map": false,
  "uses_future_truth": false,
  "auto_execute": false
}
```

注意：它不能输出 `BUILD_ROAD`，不能替 agent 执行动作，也不能读取完整地图或未来路径。
当前实现只把同类原始地形的历史 payoff 作为可行动的 `tile_specific` 证据；`global_prior` 只是背景统计，不能直接触发修路。这样可以避免把 rough road 的正收益错误泛化到普通 ground 上。

任务 8.6：Exploration-based Road Learning（已完成第一版）

目标：

```text
不再依赖 RuleRoadOracleAgent 生成初始 road records，
而是通过受控探索产生修路样本，
让 RoadLearningModule 从 agent 自身尝试中学习修路偏好。
```

8.6 不是让系统自动修路，而是让 agent 在有 `ShapingOpportunity` 时拥有探索性尝试机制。探索产生的 road payoff 会进入 `AgentMemory.road_credit_records`，随后由 `RoadLearningModule` 转成后续 episode 的 `learned_estimate`。

方案 A：epsilon exploration

```text
if ShapingOpportunity.available:
    with probability epsilon:
        BUILD_ROAD
    otherwise:
        follow route action
```

epsilon 逐渐衰减：

```text
0.30 -> 0.20 -> 0.10 -> 0.05
```

这能产生 road payoff 样本，同时避免 agent 永久乱修路。

方案 B：uncertainty exploration

优先在“有潜在收益但经验不足”的位置探索：

```text
if confidence is low
and saving_per_use > 0
and on_current_route == true:
    occasionally try BUILD_ROAD
```

这比纯 epsilon 更贴近自进化研究主题，因为它不是随机乱试，而是在 agent 已经看到 shaping opportunity、但缺少经验时尝试。

方案 C：curriculum

先用简单场景：

```text
base -- rough corridor -- ore cluster
```

再逐步增加复杂度：

```text
ground corridor
mixed ground/rough
obstacle + rough
multiple ore clusters
randomized maps
```

如果 agent 能从 rough corridor 学到“rough 高频路径值得修”，再迁移到更复杂地图，就更有说服力。

因此，8.6 第一版不能用来证明“LLM 自己学会修路”。它只能证明：

```text
在没有 oracle warm-start 的情况下，
一个非 LLM exploration-threshold agent 可以通过自身探索产生 road payoff 样本，
RoadLearningModule 可以把这些样本转化为后续修路偏好。
```

这是一条重要的工程链路，但它不是最终研究结论。

这样做是为了避免“刚修完路就在同一集里过快接管”的问题，让实验更清楚地区分：

```text
exploratory build: 为了收集样本而尝试修路
learned build: 基于过去 episode 的 road payoff 学到修路偏好
```

任务 8.7：LLM-mediated Road Learning（已完成第一版）

目标：

```text
让 LLM 接收 ShapingOpportunity、learned_estimate、uncertainty、route_context 和 exploration budget，
由 LLM 自己决定是否 BUILD_ROAD，
而不是由 ExplorationRoadAgent 的阈值规则直接决定。
```

合理对照组：

| 组别 | 说明 | 目的 |
| --- | --- | --- |
| `route_only` | 只导航，不修路 | 导航下界 |
| `exploration_threshold` | 当前 8.6 的非 LLM agent | 机制 baseline |
| `llm_no_road_learning` | LLM 看得到 ShapingOpportunity，但没有 road payoff memory | 验证 LLM 单靠提示是否会乱修 |
| `llm_with_road_learning` | LLM 看得到 ShapingOpportunity + learned_estimate + road payoff memory | 主实验 |

这样才能更接近真正的研究结论：

```text

```

### 8.8 先做 sanity check

在实现学习模块前，必须先验证：

```text
Rule-road oracle > No-road agent
```

如果手写 oracle 修路都不能提升收益，那问题不是 Agent 不会学，而是当前环境没有给修路足够价值。可能需要调整：

```text
episode 长度
矿石可重复运输次数
rough terrain 分布
move_ground / move_rough / move_road 成本差
build_road 成本
```

确认“修路确实有长期价值”之后，再让 `RoadLearningModule` 去学什么时候修路。

## 9. 不建议现在做的事

暂时不建议：

1. 直接跑 300k steps 大训练；
2. 直接让 DeepSeek 每一步控制动作；
3. 直接硬编码修路/挖掘策略；
4. 在 full-info 设定下继续堆步数；
5. 加多智能体；
6. 加仓库、桥梁、塌方等复杂机制；
7. 继续修 `text004` 里的依赖冲突。

这些都会把问题变复杂或让研究问题跑偏。当前最重要的是把信息结构改对：局部观测、自建记忆、经验反思。
